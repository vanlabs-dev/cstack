# LLM ops walkthrough: SignalGuard finding narratives

> See [docs/INDEX.md](./INDEX.md) for the full documentation map.

## Overview

cstack SignalGuard ships a deterministic CA audit (Sprint 2) and a
behavioural sign-in anomaly detector (Sprint 3). Both surface findings
into the same `findings` table. Sprint 6 adds the narrative layer: every
finding gets a triage-grade plain-English explanation generated by an
LLM and cached so identical findings across tenants reuse the same
narrative. This document is the operational view of how that pipeline
works, why it is shaped the way it is, and how the calibration loop
proves it.

The deliverable is small but opinionated: a four-package vertical slice
(`llm-provider`, `llm-narrative`, `llm-eval`, plus storage migrations)
totalling roughly 2,500 lines of Python and TypeScript. The discipline
of the sprint is producing meaningful narratives without bloating the
codebase, paying real money in eval, and writing down what we learned.

The pipeline is read-after-write at the audit boundary: the CLI command
`cstack audit all` writes findings via the normal path, then runs a
narrative pass. The API surface mirrors this: `POST /audit/run` accepts
`generate_narratives: bool` in the request body. The web dashboard uses
TanStack Query against `GET /findings/{finding_id}/narrative`, which
synchronously generates on cache miss.

## Provider abstraction

We built our own provider abstraction (one Protocol, three adapters)
rather than pulling in LiteLLM. The reasons:

- The contract is small. Three methods (`complete`, `count_tokens`,
  `supported_models`) cover everything cstack needs.
- LiteLLM's coverage is broader than we need and ships its own opinions
  about retries, rate limiting, and observability that conflict with
  ours.
- Owning the abstraction means we can ship adapter-level fixes
  immediately when, for example, a vendor deprecates a parameter (Claude
  4.7 dropping `temperature` mid-sprint, see calibration notes).

The Protocol:

```python
@runtime_checkable
class LlmProvider(Protocol):
    name: str

    async def complete(self, request: LlmRequest) -> LlmResponse: ...
    async def count_tokens(self, text: str, model: str) -> int: ...
    def supported_models(self) -> list[str]: ...
```

Three adapters: `AnthropicProvider`, `OpenAIProvider`, `OllamaProvider`.
The factory caches adapter instances per process because the underlying
SDK clients hold connection pools that should be reused. Tests register
fakes via `register_provider(name, fake)` and clear them via
`clear_registry()` in setup.

Adding a new provider is one Python file: subclass-free, just implement
the Protocol shape, register in `_build_provider`, and you are done.

### Why a Protocol, not an ABC

ABCs force inheritance. Tests would have to subclass and implement
abstract methods even when the test only cares about one method. With a
Protocol, the recording stub used in tests is a 20-line class with no
inheritance dependency. The `runtime_checkable` decorator lets opt-in
isinstance checks happen at runtime where useful.

## Prompt design

The canonical prompt template lives at
`packages/llm-narrative/src/cstack_llm_narrative/prompts/finding_narrative_v1.md`.
Frontmatter declares id, version, description, and required inputs; the
body is the actual prompt text with `{key}` placeholders.

Key design choices in v1:

- **Four required sections**: Why this fired, What it means,
  Remediation, Caveats. The generator validates the response actually
  contains all four; missing sections trigger one retry with a fix
  instruction, then raise.
- **Sentence case headings, no em dashes**: matches the project's hard
  rules. The validator enforces this.
- **Explicit length cap**: 250 words. Stops the model from drifting
  into background.
- **Prompt injection defence**: evidence is delimited by `<EVIDENCE>`
  tags and the prompt instructs the model to treat that block as data,
  not instructions. A finding's `summary` field, which is rule-author
  controlled, is not delimited because it is trusted.
- **Why temperature 0.2 not 0.0**: zero-temperature produces robotic
  identical phrasing across calls. 0.2 keeps responses consistent in
  meaning while varying surface phrasing enough to feel like a person
  wrote it. (Note: Claude opus-4-7 deprecated temperature on the
  messages endpoint; the adapter drops it for that model.)

## Caching strategy

Cache keys are content-addressed:

```
SHA-256(rule_id || canonicalised(evidence) || prompt_version || model)
```

`canonicalised(evidence)` is `json.dumps(evidence, sort_keys=True,
separators=(",", ":"))`, which makes two equivalent dicts hash
identically regardless of insertion order or whitespace. Tenant id is
deliberately excluded from the key: a `rule.block-legacy-auth` finding
with the same evidence in two tenants gets identical narratives, and
reusing a single cache entry across tenants is the highest-leverage
cost optimisation in the pipeline.

Observed hit rate during the post-calibration smoke run on tenant-b:

- First pass after calibration: 13/29 (44.8%) hit, because findings
  also present in the golden set already had cached narratives.
- Second pass on the same tenant: 29/29 (100%) hit, $0.00 spent.

LRU-by-`last_used_at` eviction. `get_cached_narrative` bumps
`last_used_at` and `use_count` on every hit so the eviction signal stays
accurate. The `cstack narrative cache-evict --days 90` command runs
the LRU sweep; the default cutoff is 90 days because narratives age out
naturally as rule-author guidance shifts.

V2 will add embedding-based semantic dedup (two findings whose evidence
is nearly identical but not byte-identical can collapse into one cache
entry). For Sprint 6, exact content-addressing is sufficient: the
fixture corpus produces enough evidence variation that pure exact
matching gives a high hit rate.

## Budget tracking

Every audit run that generates narratives passes through a
`NarrativeBudget`. Default cap: $1.00 per run, configurable via
`CSTACK_LLM_BUDGET_USD` env var or `--narrative-budget` CLI flag.

Cost estimation per call uses a hardcoded `MODEL_PRICING` table mapped
in `packages/llm-narrative/src/cstack_llm_narrative/narrative.py`. Drift
between this table and the real Anthropic bill is acceptable: budget
tracking is a guard rail, not an audited accounting figure.

When the next call would exceed budget, the generator raises
`BudgetExceededError`, the batch wrapper catches it, increments
`skipped_count`, and the run completes with a structured summary
showing how many findings have "narrative pending" status. The web UI
renders that gracefully: skeleton loader followed by the section being
absent rather than the whole expansion failing.

The lower bound of cost per narrative is the prompt size: ~500 input
tokens for a typical CA finding, ~500 output tokens for the four-section
response. At Opus 4.7 rates that is approximately $0.045 per finding;
at Sonnet 4.6 rates it is $0.009. The cache hit rate above is what
makes the system economical at scale.

## Evaluation framework

`cstack-llm-eval` ships rubric-based LLM-as-judge with explicit bias
mitigation. Five criteria, weighted:

| criterion          | weight | description                                                             |
| ------------------ | ------ | ----------------------------------------------------------------------- |
| accuracy           | 1.5    | Does the narrative reflect evidence and rule? Penalises hallucinations. |
| actionability      | 1.5    | Can an engineer act on it? Penalises hedging.                           |
| concision          | 1.0    | Is it under 250 words and on-topic? Penalises padding.                  |
| format_compliance  | 1.0    | Four sections present, sentence case, no em dashes.                     |
| tone               | 0.5    | Triage-grade engineer voice; penalises marketing tone, emoji.           |

The aggregate is a weighted mean over the [min, max] window of each
criterion (default 1-5), normalised to 0-100.

### Pointwise vs pairwise

The judge supports both:

- `score_pointwise`: rates a single narrative against the rubric. Useful
  for ranking absolute quality and tracking regressions over time.
- `score_pairwise`: picks a winner between two narratives. Useful for
  prompt-version A/B and for comparing candidate vs reference.

Both are necessary because pointwise rubric scores can be misleadingly
generous when criteria are tight: in the Sprint 6 calibration, prompt
v2 outscored v1 pointwise (98.98 vs 95.45) but lost 14-0-6 pairwise
because the harder concision floor sacrificed remediation detail. See
`docs/SPRINT_NOTES.md` for the full table.

### Bias mitigation

Three layered defences:

1. **Different judge from generator.** Default generator is
   `claude-opus-4-7`; default judge is `claude-sonnet-4-6`. A model
   asked to grade its own output inflates the score 3-8% in the LLM-judge
   literature; using a different model substantially mitigates that.
2. **Position swap on pairwise.** Every pairwise comparison runs
   twice: once with narratives in (A, B) order, once swapped. If the
   judge picks the same content in both runs, the result is reported
   with `position_swap_consistent=True`. If the judge flips its winner
   based on position, the result is reported as `tie` with consistent
   set to `False` and the inconsistency is counted in the run summary.
3. **Low temperature on judge calls.** Judge calls use
   `temperature=0.0` for maximum scoring consistency; generator calls
   use `temperature=0.2` for natural variation in phrasing. A
   high-temperature judge would re-roll its own scores and add noise
   to comparisons.

### Golden set construction

Twenty hand-curated examples sourced from real fixture findings (6
tenant-a, 8 tenant-b, 6 tenant-c). Coverage:

- All five severities (CRITICAL, HIGH, MEDIUM, LOW, INFO)
- All three categories (coverage, rule, exclusion)
- All major rule families with at least one representative

Each entry is `{finding: Finding, reference_narrative: str,
scenario_notes: str}`. The reference narratives are the human gold
standard: written to the same constraints the prompt asks the LLM to
satisfy (4 sections, sentence case, no em dashes, under 250 words).
Storage is `packages/llm-eval/data/golden_set.json`.

A reference narrative that breaks its own rules makes the eval
meaningless. All 20 references in the shipped golden set pass the
generator's `_validate_structure` check.

Full calibration run on 2026-04-29 against the live Anthropic API.

### Decision

Keep v1 as default. v2 is shipped as `prompts/finding_narrative_v2.md`
to document the experiment. The calibration loop is the value: knowing
that pointwise scores would have led us astray is more important than
either prompt individually.

### Spend

Approximately $4.30 of Anthropic API spend during calibration plus
post-calibration smoke tests, against a $20 hard cap.

## Known limitations

- **English only.** Multi-language narratives are not in scope yet.
- **No streaming.** Narratives return as a single response; UI shows a
  skeleton during the ~10-second generation window. SSE streaming is V2
  if user demand emerges.
- **Single canonical prompt template.** Anomaly findings will need
  their own template; deferred to Sprint 7+.
- **No semantic cache dedup.** Two findings with byte-different but
  semantically identical evidence get separate cache entries. Embedding
  similarity is V2.
- **No live-tenant calibration.** All eval and smoke tests use synthetic
  fixture data. Sprint 7 lands real-tenant validation.
- **Token usage logged to file, not a UI.** The `cstack narrative
  cache-stats` command is the only visible spend tracker. A spend
  dashboard is V2.
- **Custom fine-tuned models are not on the roadmap.** Once enough
  eval-passing narratives accumulate, a fine-tune is plausible, but
  the cost-quality math has to clear a steep bar before it makes sense.

## Roadmap

- **Sprint 6.5 (if user demand emerges)**: streaming narrative
  responses; spend dashboard; per-rule prompt customisation.
- **Sprint 7**: live tenant integration; calibration against real
  Graph data; prompt iteration on real-world findings.
- **Sprint 8**: anomaly narratives (consume SHAP top-3 plus rule
  references); executive tenant summaries (cross-finding aggregation).
- **V2**: embedding-based semantic cache dedup; multi-language; fine
  tune on the eval-passing corpus once it exists.
