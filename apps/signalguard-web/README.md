# signalguard-web

Next.js 15 dashboard for the cstack signalguard module. Server-component-first
with TanStack Query for client interactions; reads the FastAPI service at
[apps/signalguard-api/](../signalguard-api/README.md) via a typed client
generated by `@hey-api/openapi-ts`.

## Local dev

```sh
# 1. Hydrate fixtures into DuckDB.
uv run cstack fixtures load-all

# 2. Set the dev API key and start the API on port 8000.
echo 'SIGNALGUARD_API_DEV_API_KEY=dev-secret' >> .env
uv run signalguard-api --port 8000

# 3. In a second shell, start the web app on port 3000.
pnpm --filter signalguard-web dev
```

The first time you open `http://localhost:3000/dashboard`, the API key gate
asks for the dev key. Enter `dev-secret` (or whichever value you set in
`.env`). The key is mirrored to `localStorage` and a same-site cookie so
both server and client components can authenticate against the API on the
same value.

## Routes

| Path                                 | Description                                                                                      |
| ------------------------------------ | ------------------------------------------------------------------------------------------------ |
| `/dashboard`                         | Home: workspace greeting, module grid, tenant connections, recent activity.                      |
| `/dashboard/signalguard`             | SignalGuard tenant overview: 4 KPI cards, CA + anomaly summary, coverage preview, freshness.     |
| `/dashboard/signalguard/coverage`    | Full coverage matrix heatmap with cell drill-down side panel.                                    |
| `/dashboard/findings`                | CA audit findings with filter chips, paginated table, inline expansion, right rail.              |
| `/dashboard/anomalies`               | Anomaly feed timeline with filter rail, bulk select, paginated rows.                             |
| `/dashboard/anomalies/[signinId]`    | Sign-in drill-down with metadata table, location map, SHAP waterfall, history strip, action bar. |
| `/dashboard/settings/general`        | Workspace preferences: default tenant, theme, density.                                           |
| `/dashboard/settings/audit-rules`    | Read-only catalogue of the 15 CA audit rules with descriptions and references.                   |
| `/dashboard/settings/anomaly-tuning` | Current model summary, threshold preference, calibration metrics, retrain placeholders.          |
| `/dashboard/settings/api-keys`       | Mint and revoke per-tenant API keys.                                                             |

The Notifications, Data & sync, and Integrations tabs are placeholders
until their backend lands.

## Testing

```sh
pnpm --filter signalguard-web test          # one-shot vitest run
pnpm --filter signalguard-web test:watch    # watch mode
```

- Test files live in `__tests__/` directories adjacent to the source.
- `src/test-utils/render.tsx` provides `renderWithProviders` (QueryClient,
  fresh per test).
- `src/test-utils/fixtures.ts` holds deterministic fixtures for findings,
  anomaly scores, coverage matrices, model summaries, and tenant rows.
- recharts and `next-themes` rely on browser APIs (ResizeObserver,
  matchMedia) that jsdom does not implement; `vitest.setup.ts` stubs them.

## Responsive design

Desktop is the first-class target. The single tablet breakpoint is `md:`
(>=768px in Tailwind 4):

- Below 768px the sidebar collapses behind a hamburger toggle (see
  `MobileNav`).
- Multi-column grids fall back to single-column or 2-column layouts.
- The findings table hides ID and Age columns; the coverage heatmap
  scrolls horizontally; the coverage side panel becomes a bottom sheet.
- Phone breakpoints (<768px) render correctly but have not been visually
  audited; mobile polish is parked for a later sprint.

## Regenerating the API client

The generated client is committed at `src/lib/api/generated/`. Treat it as
read-only build output. When `apps/signalguard-api/openapi.json` changes,
regenerate:

```sh
pnpm --filter signalguard-web generate:api
```

CI re-runs this and `git diff --exit-code`s the generated tree, so a stale
client fails the build.

## Dev API key flow

- `ApiKeyGate` (`src/components/system/ApiKeyGate.tsx`) renders a single-input
  modal until a key is present in `localStorage`. Submitting writes to both
  `localStorage` and a `cstack-api-key` cookie (`SameSite=Lax`, max-age 1
  year, dev only).
- Server components resolve the key from cookies via `next/headers`; client
  components resolve from `localStorage`. Both paths share the same value.
- Clear the key from the user profile dropdown in the sidebar; the page
  reloads and the gate re-prompts.

## Design system pointer

Visual decisions live in:

- `docs/DESIGN_TOKENS.md` (single source of truth for colour, type, spacing)
- `docs/DESIGN_SYSTEM.md` (component primitives, layout shell, screen
  blueprints)

The `.design/` working folder is gitignored. Anything component code needs
should be promoted into the committed `docs/DESIGN_*.md` first.

## Tone

Sentence case throughout. No marketing language. No emoji. Severity badges
always pair colour and shape (the shape primitive is real CSS, not Tailwind
classes; see `globals.css`). All numeric tables, IDs, and timestamps use
the `mono` or `num` utility classes for tabular alignment.

## Production build

```sh
pnpm --filter signalguard-web build
pnpm --filter signalguard-web start
```

`output: "standalone"` is intentionally off (it symlinks the pnpm store,
which fails on Windows). Re-enable on Linux CI for the container deploy
sprint.
